# Failure Mode Analysis for the Multi-Agent System

## Schema and Scope
The analysis first inspected the selected result directories and uses the following schema. Each run directory contains `config.json`, `metrics.json`, `experiment.log`, and `raw_results.json`. The `raw_results.json` file is a list of per-question records with `question_id`, `question`, `prediction`, `ground_truth`, `correct`, `evaluation`, `query_analysis`, `action_history`, `output_messages`, `turns`, `tool_counts`, `token_usage`, and `metadata`. In AgentFlow/MAS records, `query_analysis` is the turn-0 orchestrator analysis; `action_history` is the compact sequence of sub-goals, tool names, commands, and sub-agent/tool results; `output_messages` stores the underlying assistant/tool transcript when present.

Scope decisions: MAS means the earliest completed Qwen3-8B `subagent_tools_*` run for each benchmark and thinking mode (`all`, `none`, `orchestrator`, `subagents`) under `experiments/results/1_milestone_no_img_no_mindmap_AgentFlow`. The baseline means the earliest completed `direct_tools_*` run under `experiments/results/NEW_baseline`, including both Qwen3-8B and Qwen3-32B and both `none` and `orchestrator` thinking modes. Counts are therefore **question-run cases**, not unique benchmark questions. Duplicate April reruns are excluded by the earliest-run rule.

## Part 1 - MAS Failure Mode Analysis
Failure modes were assigned bottom-up from observable trace patterns. The labels are mutually exclusive and applied in this order: modality/tool gap, answer-format unit error, loop/empty-final failure, no-action direct reasoning, code-driven computational error, retrieval/evidence failure, and single-shot trust. The order matters only when traces exhibit multiple symptoms; for example, an image task that also loops is counted as a modality/tool gap because the missing modality is the upstream cause.

### Direct reasoning without action
The orchestrator produced a final answer without any sub-agent/tool action even though the run condition allowed sub-agent tools. These cases are observable as empty `action_history` with a non-empty wrong prediction.

Root cause: initial query analysis and stopping decision. The orchestrator treated the first reasoning pass as sufficient and never delegated work to a sub-agent.

Sub-agent contribution: none; no sub-agent was invoked.

## Part 3 - Fine-Tuning Implications
| Priority | Failure mode | Frequency impact | Behavioral target for the orchestrator | Training signal available | Missing data / caveat |
|---:|---|---:|---|---|---|

The highest-priority fine-tuning targets are therefore stopping/action calibration (`Direct reasoning without action`), retrieval discipline, and verification after one-shot tool use. `Modality/tool-coverage mismatch` is frequent, but it should not be treated as fully recoverable by orchestrator-only fine-tuning because the experiment intentionally disabled image inspection. The cleanest small SFT target is final answer normalization: it has only two observed negative cases but unambiguous corrections.
